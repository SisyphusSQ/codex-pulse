production 首次初始化采用 `interactive` 预算和 `backfill` lane。正常状态下，同一次 robfig cron callback 只在 durable cycle 的 `YieldFor=0` 时连续推进下一 slice，不再被 1 秒 cron 精度人为插入空闲；每个 slice 边界仍重新选择 live/backfill queue 并读取 lifecycle、Home generation 与注入的 system probe。live task、用户 pause、sleep、Home fence、pressure yield 和 SQLite 单 writer 边界均未取消。该生产调度语义由 scheduler/application 集成测试证明。

真实 Home 三轮 cold bootstrap、30 次 warm query、隐私扫描、source 只读核对与隔离清理全部通过。性能 runner 为减少测量噪声，直接以 `Bootstrap Runtime.RunSlice` 的 `512MiB / 30s` 预算连续运行，不经过 production scheduler 的 `16MiB / 200ms` cycle、queue 或 cron callback；因此下列 wall time 是 core bootstrap+Store+query 基线，不冒充 packaged application 的端到端启动时间。相对 TOO-298 的同一路径正确性基线 `1,084,765ms`，本轮最坏 full-bootstrap `919,556ms`，缩短约 `15.2%`；不能与曾经延迟公共 quota projection 的错误 `9.56` 分钟候选比较。

## 已证实的优化与正确性边界

1. bootstrap 默认 reader chunk 从 1 MiB 提升到已有安全上限 4 MiB，降低 6.56 GB 首次扫描的 prefix proof 与调用开销。
2. quota projection 使用 schema v15 过滤与排序索引、dirty-window delta write/readback 与有界 stale deletion，避免每条 observation 反复全量读取累计 evidence；查询仍需回表，公开 reader 仍从 immutable observations 全量重算并 fail closed。
3. production startup 与 Home switch post-commit 都会用稳定 identity 幂等 enqueue bootstrap task；不再依赖测试或外部调用方手工入队。
4. robfig cron callback 对 `YieldFor=0` 的 durable cycle 连续 burst；每个 cycle 仍重新选队列，测试证明新 live task 可在两个 interactive backfill slice 之间抢占。
5. mutable Home 中，已冻结 source 在真正 open 前被归档、重命名或删除，归类为 `changed_during_scan` 并交给 final reconcile 重新证明；危险 symlink、Home identity 替换、权限与无法闭合的 reconcile 继续 fail closed。
6. 4,096 observation quota projection benchmark 五轮为 `19.3～20.3ms/op`；该 benchmark 记录当前分配成本，不替代真实 Home 三轮门禁。

## 失败发现与重测

- 优化探测完成 6.56 GB 后，单次 gopsutil process probe 抖动使整轮失败。采样器改为最多 10 次连续瞬时失败后才终止，并记录 retry count；第二轮的 3 次重试证明该边界可观察且会恢复。
- `performance-final-1` 三轮虽然单轮均通过，但聚合器用插值公式把三轮 p95 错算为中位数。实现改为 nearest-rank，并用三样本最差值及 30 样本第 29 位回归测试锁定；旧聚合不得作为最终证据。
- `performance-final-2` 第二轮在 6.26 GB 后因 source 在 open 前消失而失败。根因是 snapshot reader 将 `ENOENT` 原样冒泡；修复后 reader 与 bootstrap 集成测试证明 initial item 进入 drift，final reconcile 可闭合。最终 `performance-final-3` 用冻结后的实现完整重跑三轮。
- 完整 race 初次暴露 Home switch restart 的 scheduler admission 冲突。根因是 recovery 已将原 task rebind 到 resume job，而启动 helper 又按新 target 创建 task；修复为按 current target 读回稳定 admission，并覆盖 queued/running 到 terminal 的并发幂等 replay。相关 app 测试连续 10 轮与 race 均通过。
- packaged idle harness 初版向 child environment 追加了第二个 `HOME`，无法证明使用隔离 preferences；严格 durable-idle readback 以 bootstrap `missing` 阻止其成为证据。修复为去除全部旧 `HOME` 后只注入唯一隔离值，并预置 onboarding 已提交但尚未执行的 durable bootstrap；`final-support-9` 三轮均读回 `succeeded`、active task `0` 与 ownership cleanup，旧探测全部作废。
- append benchmark 初版追加未配对 completion，只形成 lifecycle diagnostic/checkpoint；独立 review 阻止其成为事实提交证据。冻结版改为 initial open turn + matching completion，并在计时外读回 canonical completed fact；support evidence 同时记录源码 tree hash 与实际 executable SHA。

## 残余边界

- 实际 runtime 是 macOS `26.5.2`，只证明 deployment target 为 15.0；没有冒充实际 macOS 15 runtime 验证。
- application 当前使用默认静态 `SystemProbe`，因此本轮“全速”表示没有人为 idle；CPU/RSS 是结果门禁，不代表生产 worker 已从 OS 动态生成 pressure bit。probe 注入、pressure budget 和逐 cycle 重读已有测试，但若后续要按真实系统压力自动降档，应建立独立 Issue 明确采样窗口与防振荡 contract。
- 三轮真实 Home runner 不经过 production scheduler；本卡用生产装配、durable enqueue/rebind、interactive zero-yield burst、background completed cooldown 与 live 抢占测试证明调度 contract，但没有给 packaged app 的首次初始化单独测 wall time。若 PER-01 后续要求产品端到端数值，应在不暴露真实 Home 的前提下增加真实 application runner，不能复用本表数字冒充。
- 本卡不覆盖 WebView paint、AppKit 窗口视觉、VoiceOver 或多显示器交互；这些仍由 `TOO-301` 的人工 macOS 矩阵负责。
- GitHub Actions 依用户要求保持关闭；所有结论来自本地门禁。
