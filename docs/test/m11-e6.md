## 执行副作用与隔离

`CODEX_PULSE_RUN_M11_UPGRADE_E2E=1 make m11-upgrade-recovery` 会：

- 编译三个 arm64 ad-hoc App binary，并生成 source/current/failure candidate bundle。
- 在进程内生成一次性 synthetic Ed25519 seed，经 stdin 交给官方 Sparkle tooling；seed 不进入 argv、环境、日志、evidence 或 artifact。
- 只在随机 `127.0.0.1` 端口提供临时 appcast/archive，使用每场景唯一 synthetic bundle id。
- 在系统临时目录创建 `0700` scenario root、`0600` control/DB/preferences/evidence；不读取 Codex Home、auth、用户数据库或生产 preferences。
- 通过 LaunchServices 启动真实 bundle，按 executable identity 追踪/终止进程；成功后删除临时 root、defaults 和本轮新增的 Sparkle cache entry。失败只保留 content-free 诊断 root，再按同一 ownership/identity 规则清理。

## 失败、恢复与重跑

| 失败点 | 停止证据 | 恢复方式 |
| --- | --- | --- |
| choice / drain | 首个 failing test 与 typed stage | 保留 durable intent/首次 terminal fault；修复后重跑 UPD-01/02 focused |
| disk/read-only/backup | `space` 或 `backup` typed code；未进入 apply | 恢复空间/权限或 backup adapter，确认旧 schema/ledger未推进 |
| signature/offline | `invalid_signature` 或 `check` | 保留 source bundle/DB；修复 fixture后只重跑对应场景 |
| migration apply | `migration_safe_mode/apply_failed` + backup boolean | restore N-1 bundle/backup，验证 schema 14和marker，再重跑 migration_failure与success |
| cleanup | synthetic进程/default/cache/root残留 | 停止 closeout，按 exact executable/bundle id/entry ownership清理并完整重跑 |

重跑必须显式 opt-in。成功运行是幂等的，每次使用新临时 root、随机端口和唯一 bundle id；不复用正式 key或生产 appcast。
