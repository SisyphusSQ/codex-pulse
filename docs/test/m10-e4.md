## 验证目标

1. Quit 与 Install 共用单一 `running -> draining -> closed` 状态机；并发调用幂等，timeout 后可以重等但不能恢复 admission。
2. instance wake/updater/scheduler admission、Tray/retention/health、scheduler drain、metrics、SQLite/instance lease 按依赖顺序关闭；任一失败可观察且阻止 Sparkle final reply。
3. scheduler/lifecycle 先封闭新 job，协作式等待当前 slice/checkpoint；SQLite Store drain admitted writer，不向事务注入强杀。
4. `readyToInstall` 只有经过用户确认和 safe close 后才消费 Sparkle 的 `installChoiceReply`。
5. 第二实例只发送 wake 并退出；owner 显式释放后可 takeover，崩溃后 flock 可回收，lock inode 不删除。

## 状态与失败矩阵

| 场景 | 预期 |
| --- | --- |
| 正常 Quit | shared shutdown 完成后才调用 Wails Quit；Wails hooks 幂等释放 Sparkle/Tray |
| 正常 Install | UI 二次确认；显示 draining；safe close 成功后 final reply 恰好一次 |
| Install 与 Quit/Cmd+Q 并发 | Install 独占 terminal intent；native dispatch 前拒绝 Quit，`install_started` 回写后才允许由 final reply 触发 terminate |
| drain timeout | 返回当前 stage；不调用 install；后台继续；重试只等待同一 close |
| 组件失败 | 记录 `failedStage`，继续清理后续组件，但不安装 |
| 第二实例 | 固定 wake frame 唤醒主窗口，第二进程成功退出且不打开 SQLite |
| owner 正在退出 | contender 交替尝试 wake/同 inode flock，最终 wake 或合法接管 |
| crash/stale socket | 内核释放 flock；新 owner 持锁后才删除 stale socket并监听 |
