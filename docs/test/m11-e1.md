## 目标

- 以稳定场景 ID 连接 Onboarding、索引、账本、Quota、UI、Tray、Health 和 Update 全链路。
- 每项明确执行类型、前置/数据、操作与预期、提交版证据、清理、副作用、责任卡和当前结果。
- 把正常、空态、加载/取消、重复事件、依赖失败、恢复和关键边界纳入最终 gate。
- required live E2E 缺环境或未执行时保持 blocker，不用 fixture 或历史推断替代。

## 执行副作用

- DB：TOO-297 不打开用户数据库；后继真实数据/E2E 只允许独立临时 DB 或明确批准的应用私有数据库，并必须记录备份和清理。
- 服务：本卡不启动临时服务；更新 E2E 若使用 loopback feed，由 TOO-302 负责启动、停止和残留检查。
- 文件：允许 Go/npm/Wails 构建缓存和 ignored `bin/`、`frontend/dist/`；原始输出只放 `.artifacts/runs/<RUN_ID>/`，不提交。
- 外部系统：不触达 GitHub Actions、release API、在线 quota 或其它第三方系统。
- 用户数据：本卡不读取 `~/.codex`；TOO-298 必须只读、metadata-only、隔离数据库并在执行前再次声明范围。

## 阻塞、失败回写与重测规则

1. 仍在用户范围内的 required 场景为 `失败`、`blocker` 或 `未执行` 时，对应责任 Execution 不得进入 Done；`EXCLUDED` 只允许引用用户明确取消范围，不代表测试通过，也不阻止本次 M11/Root closeout。
2. 失败回写必须包含场景 ID、当前 commit、执行类型、typed 失败、影响、清理状态、恢复条件和责任卡；不得贴凭据或原始用户内容。
3. 修复后先重跑失败场景与直接回归集；只有改变共享 contract、schema、lifecycle 或 release链时才升级为跨域重跑。
4. 同类 finding 连续两轮出现时转 architecture/root-cause review，不进入第三轮同类补丁。
5. Actions disabled 不是 blocker，但必须记录 `actions_disabled_by_user`，不能宣称 CI 通过。
6. 正式发布缺授权时 REL-01 的 artifact readiness 可以通过，外部分发动作保持“未授权/未执行”，不得伪装成已发布。

## 清理与提交版证据边界

- 每个 live run 必须证明：应用/loopback server停止、测试 DB/fixture按策略清理、Codex Home 未修改、Preferences/cache无本轮 synthetic残留。
- 允许提交：场景 ID、版本/commit、typed code、计数、分位数、schema version、布尔检查、脱敏截图和清理结论。
- 禁止提交：Authorization/Cookie/token/auth内容、原始 JSONL、prompt/response/tool output、完整 URL、绝对路径、临时目录、PID日志、server log和完整 transcript。
- 原始输出进入 ignored `.artifacts/runs/<RUN_ID>/`；提交版只保留本文件或责任卡 runbook 中的脱敏摘要。

## 失败处理

| 失败点 | 停止条件 | 恢复方式 |
| --- | --- | --- |
| E1 静态 gate | 缺场景/责任/边界/引用 | 修正文档或引用后重跑，不继续 review |
| 自动化或 package | 任一命令非零 | 保留脱敏摘要，清理构建产物，修复后按漏斗重跑 |
| 真实数据/平台 E2E | 环境、权限、隐私范围不明 | 停在 blocker，明确所需条件，不用 fixture替代 |
| 清理 | 进程、Preferences、cache、DB或fixture残留 | 停止并完成精确清理后才允许下一场景 |
| 正式发布 | 未获得独立授权 | 停在 release-readiness，不执行外部副作用 |
