## 真实架构与隔离边界

orchestrator 一次编译三个仅带 `upgradee2e` tag 的 app binary：

1. `0.1.0/100` source：使用真实 migration catalog v1～v13。
2. `0.2.0/200` candidate：使用完整 catalog v1～v14。
3. `0.2.0/200` failure candidate：只在专用 tag 中让 v14 GORM transaction 完成 apply 后返回 injected error，验证 transaction rollback。

普通 `production sparkle` 构建只编译无行为 stub，不能读取 E2E control、改变路径、自动确认更新或注入 migration failure。fixture control 固定与安装目录同级，要求 `0600` regular file、canonical root 一致、全部路径位于私有 `0700` root；database、preferences、runtime、evidence 与结果均在该 root 内。

每次运行还会基于临时 scenario root 注入唯一的 synthetic `CFBundleIdentifier`，同场景 source/candidate/rollback 保持一致、不同运行互不共享。orchestrator 登记全部 synthetic id，通过 `defaults delete` 与 exact-path 清理关闭 `cfprefsd` 异步回写窗口；对 `Sparkle_generate_appcast` 只删除本轮新增的 hash entry，保留运行前已有条目。成功前必须通过真实 HOME 前后快照 gate，失败路径也执行清理。

启动入口在 `open -n -W` 后按 bundle 内真实 executable 绝对路径取得并核验 app PID；所有 return 路径都有 identity-checked cleanup。首条 evidence 前卡死仍能终止真实 app，relaunch/rollback 则以最新 evidence PID 补充追踪；exit timeout 会终止匹配 executable 的 PID。首轮与 rollback application log 分文件保留。

每个场景通过 LaunchServices `open -n -W` 启动真实 ad-hoc app。feed 只监听随机 `127.0.0.1` 端口；只有 fixture plist 注入 loopback HTTP ATS 例外，生产 release/build 继续要求 HTTPS。candidate ZIP 由 `ditto --keepParent` 生成并由官方 Sparkle 2.9.4 下载、EdDSA 验签、解压、替换和 relaunch。

```bash
CODEX_PULSE_RUN_UPGRADE_E2E=1 go run ./scripts/sparkle/upgradee2e -scenario all
```

收敛后结果：

| 场景 | 真实路径 | PASS 证据 |
| --- | --- | --- |
| success | manual check → update found → 用户 download confirm → bytes/extract → ready → install confirm → shared safe drain → Sparkle replace/relaunch → startup v13→v14 migration | target version `0.2.0/200`、schema 14、v14 前 backup 存在、source marker session 可读、target PID 退出 |
| bad signature | appcast 签名后篡改 ZIP bytes → real download → Sparkle underlying `SUSignatureError/SUValidationError` | Go typed fault `invalid_signature`；source 仍为 0.1.0/schema 13，marker 保留，未安装 candidate |
| offline | feed 指向已关闭的 loopback port → manual check | checking 阶段的 native `SUDownloadError` 按状态上下文归一为 `check`；marker 保留，无安装 |
| information-only | 已由 Ed25519 签名 enclosure 的 appcast 标记 `informationalUpdate` 并提供 HTTPS info URL | native bridge 保留 info-only/URL；Go snapshot 进入 available，Go/native 双层拒绝 Download，前端只提供系统浏览器详情动作并 dismiss/snooze 确认 |
| migration failure | 正常下载/验签/安装/relaunch → target v14 transaction injected failure | recovery graph 为 `apply/apply_failed`，v13 前 backup 存在；恢复保存的 N-1 bundle 后 schema 仍为 13、marker 可读 |

失败运行保留 `/var/folders/.../codex-pulse-upgrade-e2e.*` 并打印路径；成功运行删除 root。evidence 只包含 scenario、app version、stage、typed code、schema、marker/backup boolean 和 PID，不包含用户内容、URL、路径、SQL 或 secret。

本次最终统一入口依次打印 `build-test-package`、`synthetic-sign-appcast-verify`、`n-minus-one-upgrade`、`complete`，五个 live 场景均打印 `PASS`。初次未设置 Wails PATH 的尝试在当时的架构检查阶段、执行构建前按 `TOOLCHAIN-001` fail closed；补齐当前机器已安装的 CLI 路径后完整通过。

## 真实 E2E 发现并闭合的生产问题

1. `checkForUpdateInformation` 只有 information delegate，不能建立自定义 user-driver 的下载选择 reply；已改为 `checkForUpdates`。
2. delegate 先发布 `update_found` 会早于 user driver 保存 reply；已改为保存 reply 后才发布 availability，立即确认下载不再竞态失败。
3. Sparkle 将坏签名包装在外层 `SUInstallationError`；bridge 有界解包 underlying `SUSignatureError/SUValidationError` 后归一为 `invalid_signature`。
4. feed 网络失败使用通用 `SUDownloadError`；controller 在 `checking` phase 归一为 `check`，真正 archive download 仍为 `download`。
5. Sparkle 可能从 user driver 与 delegate 发重复终态；state machine 保留首次 terminal fault，迟到 wrapper 不覆盖可归因错误。
6. `SPUUserUpdateStageDownloaded/Installing` 不能从 checking 直接跳到 installing；bridge 现在原子形成 `available + readyToInstall`，等待共享 safe drain 后才回复 Sparkle Install。
7. information-only fallback 不能回复 Install；bridge 现在保留 HTTPS info URL，controller 与 native adapter 双层拒绝 Download，UI 只提供外部详情/确认流。
8. 唯一 bundle id 不等于 HOME 隔离；orchestrator 增加 defaults/cache 清理、`cfprefsd` 稳定窗口和前后快照 gate。
