# M9-E2 Wails3 Tray、Popover 与 NSStatusItem 能力探针

关联 Issue：`TOO-286`

## 验证目标

- 在 macOS 15+ arm64 上精确验证 `Wails v3.0.0-alpha2.117` 的 template Tray、左/右键、native menu、attached window、显隐/聚焦与 `NSStatusItem` 相对定位。
- 读回 repo-local capability matrix 和 content-free JSON evidence，明确 public API、实现内证据、已知缺口与 fallback。
- 不把 Wails attached WebView window 误称为 `NSPopover`，不把私有 `NSStatusItem` handle/custom view 当作公开能力。

## 执行副作用与输出

- DB / 用户数据：无；不打开 Codex Home、应用 SQLite、Preferences、Keychain 或凭据。
- UI：短暂出现 `Codex Pulse Tray Capability Probe` 菜单栏图标和 360×240 frameless window；硬超时为 3 秒～2 分钟。
- 文件：clean package 生成 ignored `bin/`；probe 只把 content-free JSON 原子写入 ignored `.artifacts/runs/too-286/evidence-<UTC>-<shell-pid>.json`，权限固定为 `0600`。
- 进程：只启动本轮 probe PID，结束后验证 PID 消失；不终止已有同名进程。
- 外部系统：无；GitHub Actions 保持 disabled，不查询、不触发、不等待；不发布。
- 生命周期：Wails macOS 退出不保证 `App.Run()` 返回；evidence 必须在同步 `OnShutdown` 写入，返回后写入只作为幂等防线。

### 3. Bounded live probe

```bash
mkdir -p .artifacts/runs/too-286
evidence_path=".artifacts/runs/too-286/evidence-$(date -u +%Y%m%dT%H%M%SZ)-$$.json"
test ! -e "$evidence_path"
go build -o .task/trayprobe ./cmd/trayprobe
.task/trayprobe \
  --icon "bin/Codex Pulse.app/Contents/Resources/codex-pulse-tray-template.png" \
  --evidence "$evidence_path" \
  --duration 12s &
probe_pid=$!
cleanup_probe() {
  if kill -0 "$probe_pid" 2>/dev/null; then
    kill "$probe_pid"
    wait "$probe_pid" 2>/dev/null || true
  fi
}
trap cleanup_probe EXIT INT TERM
wait "$probe_pid"
probe_status=$?
trap - EXIT INT TERM
if [ "$probe_status" -ne 0 ]; then
  exit "$probe_status"
fi
if kill -0 "$probe_pid" 2>/dev/null; then
  echo "probe PID still running after wait" >&2
  exit 1
fi
```

在 12 秒窗口内执行一次左键和一次右键，并点击 `Probe action`。如果自动化环境没有 Accessibility/UI 输入权限，仍必须完成自动定位、显示、聚焦、隐藏和 bounded shutdown；真实点击回调转入 M9-E6 的受控人工/多显示器验证，不得伪造 evidence。

JSON 最少应包含：

- `wails_version=v3.0.0-alpha2.117`、`goos=darwin`、`goarch=arm64`；
- `template_icon_configured`、`attached_window_configured`、`app_started`；
- `screen_inventory`；本机只有一块显示器时必须记录真实 count，不伪造多显示器结论；
- `position_window_call_succeeded` 与 `window_geometry_readback`；前者不得单独解释为 anchor PASS；
- `window_show/window_focus/window_visible_readback/window_hide/window_hidden_readback`；
- `app_shutdown`；
- 若真实 UI 输入成功，额外包含 `left_click/right_click/menu_action`，否则明确记录为 M9-E6 remaining gate。

读回：

```bash
test "$(stat -f %Lp "$evidence_path")" = "600"

```

## 清理

进程清理由启动 probe 的同一个 shell 直接启动 `.task/trayprobe` binary，并通过 trap 持有该 exact child 身份；不经 `go run` wrapper、不持久化 PID、不使用进程名批量终止，避免 orphan child 或 stale PID 复用误杀。验证完成后可执行 `wails3 task package:clean` 清理 ignored package 产物。

## 当前结论与 fallback

| 能力 | 结论 | fallback / 后续门禁 |
| --- | --- | --- |
| template icon、左/右键、menu | public API supported | UI callback 在本机可交互时记录；最终真实交互由 M9-E6 再覆盖 |
| attached window、显隐/聚焦 | supported | 失败时打开 regular main window |
| `NSStatusItem` 相对定位 | `adapter-required`；public call + geometry readback | `orderFrontRegardless` 有显示副作用；Retina offset 重复缩放已复现；多显示器实际几何由 M9-E6 覆盖 |
| `NSStatusItem` handle/custom view | adapter-required | 隔离 AppKit adapter 或预渲染 template fallback |
| native `NSPopover` | adapter-required / 非必需 | 优先 frozen frameless attached window |

首次 live probe 还发现 Wails macOS 退出不保证 `App.Run()` 返回；把 JSON 从 Run-return 路径迁入同步 `OnShutdown` 后读回成功。shutdown 期间 Wails 会输出 `Window #1 not found` warning，但本轮 PID、窗口与 status item 均按时退出，未产生 listener 或残留进程；该 warning 作为锁定 alpha 的已知缺陷保留，不转译为产品错误。

几何读回确认锁定 alpha 的水平 centering 可用，但不能直接采用请求 offset：status item 底边为 `31pt`，window 顶边为 `49pt`，请求 `5` 最终形成 `18pt` gap；这与实现中把 offset 乘 Retina scale 后再写 AppKit point 坐标的风险一致。生产 adapter 必须先以零 offset 校准，仍漂移则回退 regular main window；本卡不修改上游 Wails 私有实现。
