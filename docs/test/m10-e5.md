## 验证目标

1. migration gate 在任何业务 service/scheduler/writer 对外开放前执行。
2. pending user schema 先检查空间并创建可定位、`0600`、包含 committed WAL 的备份；fresh/current 跳过。
3. apply/verify 失败不推进 `user_version`，应用进入仅开放诊断与恢复命令的安全模式。
4. retry、prepare/confirm restore、restart/exit 均有稳定状态、single-flight、取消和 content-free audit。
5. restore 先生成并验证新副本，再保留失败库后原子切换；不得覆盖唯一副本或留下 canonical path 缺口。

## 故障矩阵

| 场景 | 预期 |
| --- | --- |
| fresh/current | 不创建备份；验证通过后才启动 normal graph |
| pending migration | 空间检查 -> backup -> atomic apply -> verify -> normal graph |
| newer/drift | 不尝试 apply；safe mode 展示 stable stage/code |
| disk full/backup failure | 不进入 apply；保留数据库和 partial cleanup 证据 |
| apply/verify failure | transaction rollback，`user_version` 与 ledger 不推进；safe mode |
| retry success | 进入 `restart_required`；不热装配 normal graph |
| restore 未确认/重复 token | fail closed；不创建或切换文件 |
| restore verify failure | canonical DB 不变；临时文件可定位或安全清理 |
| switch 第二步失败 | preserved DB 回滚为 canonical；继续 safe mode |
| 并发 retry/restore | single-flight；最多一个 mutation owner |
